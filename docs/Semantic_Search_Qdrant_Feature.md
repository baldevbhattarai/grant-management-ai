## Executive Summary

The current Q&A chatbot uses SQL `LIKE` keyword matching to find relevant report sections. This means synonyms, paraphrasing, and conceptually similar content are missed entirely — searching "better quality" finds nothing when the report says "improved outcomes."

This document describes upgrading the chatbot's retrieval layer to **semantic vector search using Qdrant**, enabling true natural language understanding. The solution is:

- **Fully local** — embeddings generated by Ollama, no external API calls required
- **Configurable** — supports Qdrant Cloud (web-hosted) or local Docker via a single config switch
- **Non-breaking** — existing SQL keyword search kept as fallback

## Problem Statement

### Current Behavior (Keyword Search)

```
User asks: "How many patients received better quality care?"
Keywords extracted: ["patients", "better", "quality", "care"]
SQL LIKE search: WHERE ResponseText LIKE '%better%' OR LIKE '%quality%'
Result: No match — report says "improved patient outcomes and service delivery"
```

### Expected Behavior (Semantic Search)

```
User asks: "How many patients received better quality care?"
Embedding model converts question → 384-dimensional vector
Qdrant finds sections with highest cosine similarity
Result: "improved patient outcomes" — 0.89 similarity score ✅
         "enhanced care delivery"   — 0.84 similarity score ✅
         "telehealth services"      — 0.71 similarity score ✅
```

### Impact of Current Limitation

| Scenario | Keyword Result | Semantic Result |
|----------|---------------|-----------------|
| "better" vs "improved" | ❌ Miss | ✅ Match |
| "patients" vs "people served" | ❌ Miss | ✅ Match |
| "telehealth" vs "remote care" | ❌ Miss | ✅ Match |
| "challenges" vs "barriers faced" | ❌ Miss | ✅ Match |
| "funding" vs "financial support" | ❌ Miss | ✅ Match |

### Detailed Analysis

#### Pinecone
- **Pros:** Fully managed, excellent performance, easy API
- **Cons:** Cloud-only (no local option), proprietary, costly at scale, data leaves your infrastructure — **disqualified for HRSA data sensitivity requirements**

#### Weaviate
- **Pros:** Rich GraphQL API, built-in ML modules
- **Cons:** Heavy resource footprint, complex setup, GraphQL adds learning curve for .NET team

#### Chroma
- **Pros:** Extremely easy setup, Python-native
- **Cons:** No official .NET SDK, primarily designed for Python/research use, not production-grade for enterprise

#### pgvector
- **Pros:** Runs inside PostgreSQL, familiar SQL interface
- **Cons:** Requires PostgreSQL — our stack uses SQL Server which has no equivalent. Migrating databases is out of scope.

#### Milvus
- **Pros:** Highly scalable, good for very large datasets
- **Cons:** Complex Kubernetes-based deployment, overkill for our data volume (~200 sections), limited .NET support

## Why Qdrant

Qdrant is the best fit for this project for the following reasons:

### 1. Only option that works both locally and cloud-hosted
Qdrant Cloud provides a free 1GB cluster (sufficient for thousands of report sections) with the **exact same API** as the local Docker version. Switching between environments requires changing one config value — no code changes.

### 2. Official .NET SDK
`Qdrant.Client` is an officially maintained NuGet package. No community workarounds or raw HTTP calls needed.

### 3. Rich filtering alongside vector search
Qdrant supports payload filters — we can search semantically AND filter by `grantId`, `userId`, `reportYear` simultaneously:
```json
{ "filter": { "must": [{ "key": "grantId", "match": { "value": "abc-123" } }] } }
```
This is critical for security — users must only see their own grant data.

### 4. Ollama native compatibility
Qdrant accepts any embedding vector. Ollama's `nomic-embed-text` model produces 768-dimensional vectors that drop straight into Qdrant with no adapter needed.

### 5. Lightweight local deployment
Single Docker command, ~200MB image, runs comfortably alongside the existing .NET API and Ollama on a developer laptop.

### 6. Apache 2.0 license
No licensing concerns for HRSA/government project use.

## Architecture Design

### Component Overview

```
┌─────────────────────────────────────────────────────┐
│                   Angular Frontend                   │
└─────────────────────┬───────────────────────────────┘
                      │ POST /api/ai/chat
┌─────────────────────▼───────────────────────────────┐
│              .NET 9 API (ChatbotController)          │
└─────────────────────┬───────────────────────────────┘
                      │
┌─────────────────────▼───────────────────────────────┐
│              ChatbotService (upgraded)               │
│                                                      │
│  1. Embed question via IEmbeddingService             │
│  2. Search Qdrant via IVectorSearchService           │
│  3. Build prompt with retrieved sections             │
│  4. Call AI (Ollama/Claude) via IOpenAIService       │
└──────┬──────────────────────────┬───────────────────┘
       │                          │
┌──────▼──────┐          ┌────────▼────────┐
│ Ollama      │          │ Qdrant          │
│ (Embeddings)│          │ (Vector Store)  │
│ nomic-embed │          │ Cloud or Docker │
└─────────────┘          └─────────────────┘
```

### New Services

| Service | Interface | Responsibility |
|---------|-----------|---------------|
| `OllamaEmbeddingService` | `IEmbeddingService` | Generate float[] vectors from text using Ollama |
| `QdrantVectorService` | `IVectorSearchService` | Upsert and search vectors in Qdrant |
| `VectorIndexingService` | — | Background service to index report sections on save |

### Qdrant Collection Schema

**Collection name:** `report_sections`

| Field | Type | Description |
|-------|------|-------------|
| `id` | UUID | Maps to `ReportSection.SectionId` |
| vector | float[768] | `nomic-embed-text` embedding of `ResponseText` |
| `grantId` | string (payload) | For filtering by grant |
| `userId` | string (payload) | For user-scoped security |
| `reportId` | string (payload) | Report reference |
| `reportingYear` | int (payload) | For date filtering |
| `reportingQuarter` | string (payload) | e.g. "Q1" |
| `sectionName` | string (payload) | e.g. "PerformanceNarrative" |
| `responseText` | string (payload) | Original text (returned with results) |

## Configuration Design

### appsettings structure

```json
{
  "AI": {
    "Provider": "Ollama",
    "Ollama": {
      "BaseUrl": "http://localhost:11434/v1",
      "Model": "qwen2.5-coder:7b",
      "EmbeddingModel": "nomic-embed-text"
    },
    "Claude": {
      "ApiKey": "",
      "Model": "claude-haiku-4-5-20251001"
    },
    "VectorSearch": {
      "Provider": "Qdrant",
      "Qdrant": {
        "Mode": "Cloud",
        "Cloud": {
          "Url": "https://your-cluster.qdrant.io",
          "ApiKey": "your-qdrant-cloud-key",
          "CollectionName": "report_sections"
        },
        "Local": {
          "Url": "http://localhost:6333",
          "CollectionName": "report_sections"
        }
      }
    }
  }
}
```

### Switching between Cloud and Local

Change only one value:

```json
"Mode": "Cloud"   // → uses Qdrant Cloud
"Mode": "Local"   // → uses local Docker
```

No code changes required. The `QdrantVectorService` reads the mode at startup and connects accordingly.

## Data Flow

### Indexing (one-time + on save)

```
Report section saved/updated
        ↓
VectorIndexingService triggered
        ↓
OllamaEmbeddingService.EmbedAsync(responseText)
→ POST http://localhost:11434/api/embeddings
   { "model": "nomic-embed-text", "prompt": "..." }
← float[768] vector
        ↓
QdrantVectorService.UpsertAsync(sectionId, vector, payload)
→ PUT https://qdrant-host/collections/report_sections/points
        ↓
Vector stored with grantId/userId/sectionName payload
```

### Search (on every chat question)

```
User question: "How many patients got better care?"
        ↓
OllamaEmbeddingService.EmbedAsync(question)
← float[768] question vector
        ↓
QdrantVectorService.SearchAsync(vector, grantId, topN: 3)
→ POST /collections/report_sections/points/search
   { "vector": [...], "filter": { "grantId": "..." }, "limit": 3 }
← [ { score: 0.89, payload: { responseText: "...", sectionName: "..." } } ]
        ↓
Top results used as context for AI prompt
        ↓
Answer returned with source citations
```

## Implementation Steps

### Phase 1 — Qdrant Cloud Setup (no code)
- [ ] 1.1 Create free Qdrant Cloud account at cloud.qdrant.io
- [ ] 1.2 Create cluster, get URL and API key
- [ ] 1.3 Add credentials to `appsettings.Development.json` (gitignored)

### Phase 2 — Embedding Service
- [ ] 2.1 Pull `nomic-embed-text` model in Ollama: `ollama pull nomic-embed-text`
- [ ] 2.2 Add `IEmbeddingService` interface to `GrantManagement.Core`
- [ ] 2.3 Implement `OllamaEmbeddingService` in `GrantManagement.Services`
  - Calls `POST /api/embeddings` on Ollama
  - Returns `float[]`
- [ ] 2.4 Unit test: verify embedding returns correct dimensions (768)

### Phase 3 — Vector Search Service
- [ ] 3.1 Add NuGet package: `Qdrant.Client`
- [ ] 3.2 Add `IVectorSearchService` interface to `GrantManagement.Core`
- [ ] 3.3 Implement `QdrantVectorService` in `GrantManagement.Services`
  - Reads `AI:VectorSearch:Qdrant:Mode` from config
  - Connects to Cloud or Local accordingly
  - `CreateCollectionIfNotExistsAsync()` — idempotent startup
  - `UpsertAsync(id, vector, payload)`
  - `SearchAsync(vector, grantId, topN)` — filtered by grantId
- [ ] 3.4 Register services in `Program.cs`

### Phase 4 — Index Existing Data
- [ ] 4.1 Create `VectorIndexingService` — indexes all existing `ReportSections` with non-null `ResponseText`
- [ ] 4.2 Run as a one-time startup task (skips already-indexed sections)
- [ ] 4.3 Hook into `ReportRepository.UpdateSectionAsync` to re-index on save

### Phase 5 — Upgrade Chatbot
- [ ] 5.1 Replace `SearchSectionsAsync` (SQL LIKE) in `ChatbotService` with `IVectorSearchService.SearchAsync`
- [ ] 5.2 Remove keyword extraction — no longer needed
- [ ] 5.3 Update prompt builder to use vector search results
- [ ] 5.4 Keep SQL LIKE as fallback if Qdrant is unreachable (`try/catch` → fallback)

### Phase 6 — Testing
- [ ] 6.1 Manual test: synonym queries (better/improved, patients/people)
- [ ] 6.2 Manual test: switch Mode from Cloud to Local and back
- [ ] 6.3 Unit test: `QdrantVectorService` with mocked Qdrant client
- [ ] 6.4 Verify grantId filtering (user cannot see other grants' data)

### Phase 7 — Local Docker Setup (future)
- [ ] 7.1 Document Docker command: `docker run -p 6333:6333 qdrant/qdrant`
- [ ] 7.2 Test switching `Mode` to `Local`
- [ ] 7.3 Verify data parity between Cloud and Local

## Token Usage & Efficiency

This is a critical consideration — semantic search directly impacts how many tokens are sent to the AI on every chat request.

### How Semantic Search Reduces Token Usage

The AI prompt has two parts: **context** (retrieved sections) + **question**. The quality of retrieved sections determines how many tokens are wasted on irrelevant content.

#### Current (keyword search) — inefficient
```
Keywords: ["patients", "better", "quality"]
→ Retrieves 2 sections with partial matches (low relevance)
→ Both sent to AI even if mostly irrelevant
→ AI has to sift through noise to find the answer
→ Prompt: ~600 tokens, answer quality: poor
```

#### With semantic search — efficient
```
Question embedding → similarity search
→ Retrieves top 2 sections with score > 0.75 (high relevance)
→ Only truly relevant content sent to AI
→ AI answers directly from clean context
→ Prompt: ~400 tokens, answer quality: high
```

### Token Optimization Strategy

| Technique | Description | Token Saving |
|-----------|-------------|-------------|
| **Score threshold** | Only include sections with similarity > 0.75, skip low-relevance results | 20-40% |
| **Snippet truncation** | Send top 300 chars of each section, not full text | 30-50% |
| **topN = 2** | Retrieve only 2 most relevant sections (not 5) | 40-60% |
| **No keyword extraction** | Removes the keyword-extraction step entirely, prompt is cleaner | Minor |

### Embedding Cost

Embeddings are generated locally via Ollama — **zero API cost, zero tokens consumed** from the AI quota.

| Operation | Model | Cost |
|-----------|-------|------|
| Generate embedding (indexing) | `nomic-embed-text` (Ollama) | Free — local |
| Generate embedding (search) | `nomic-embed-text` (Ollama) | Free — local |
| AI answer generation | `qwen2.5-coder` (Ollama) | Free — local |
| AI answer generation | Claude Haiku (if switched) | ~$0.0003 per query |

### Estimated Tokens Per Chat Query (After Upgrade)

| Component | Tokens (approx) |
|-----------|----------------|
| System prompt | ~30 |
| 2 retrieved sections × 300 chars | ~150 |
| User question | ~20 |
| **Total prompt** | **~200** |
| AI answer | ~150 |
| **Total per query** | **~350** |

Compare to current: ~600 tokens per query — **~40% reduction** with better answer quality.

### Re-indexing Cost

| Scenario | Embeddings Generated | Time (approx) |
|----------|---------------------|---------------|
| Initial index (175 sections) | 175 | ~5 seconds |
| Single section update | 1 | <0.1 second |
| Full re-index (future — 1000 sections) | 1000 | ~30 seconds |

All embedding generation is local — no rate limits, no API costs.

## Risk & Mitigation

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| Qdrant Cloud outage | Low | Medium | SQL keyword fallback if Qdrant unreachable |
| `nomic-embed-text` not pulled | Medium | High | Startup check with clear error message |
| Embedding dimension mismatch | Low | High | Collection created with fixed size (768), validated at startup |
| Re-indexing performance (175 sections) | Low | Low | One-time batch at startup, ~2-3 seconds total |
| Docker not available for local option | Low | Low | Cloud option works without Docker |
